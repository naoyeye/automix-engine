一、总体目标拆解（先对齐认知）
---------------

你要做的不是普通播放器，而是一个 **自动 DJ Mixing Engine**，核心能力包括：

1.  **音频解码**（mp3 / flac / dsd）
2.  **音频特征分析**
    *   节拍 / BPM
    *   拍点（beat grid）
    *   音色 / 谐波 / 频谱特征
    *   调性（Key）
    *   能量走势（适合进 / 出）
3.  **歌曲相似度建模**
4.  **过渡策略生成**
    *   对齐拍点
    *   时间拉伸（time-stretch）
    *   调性匹配（key match）
    *   Crossfade / EQ / Filter
5.  **实时音频播放与混音**

可以抽象为一条 pipeline：

```
音频文件 → 离线分析 → 特征库
                     ↓
                 播放决策
                     ↓
              实时音频混合输出
```

二、推荐开发路径（分阶段，别一口吃胖子）
--------------------

### 阶段 1：基础播放器 + 离线分析（MVP）

#### 1️⃣ 音频解码层

**不要自己写解码器**，直接用成熟库：

*   MP3 / FLAC
    *   `FFmpeg`
    *   `libmpg123`
    *   `libsndfile`
*   DSD（重点坑）
    *   需要先 **转 PCM**
    *   常见方案：
        *   `libdsd`
        *   FFmpeg（dsd → pcm\_f32le）

👉 **统一输出格式**：

> PCM float32 / 44.1k 或 48k / stereo

这是后面所有分析 & mixing 的基础。

* * *

#### 2️⃣ 音频特征提取（核心）

建议 **离线分析 + 缓存结果**，不要边播边算。

##### 常见特征维度

| 类别 | 特征 |
| --- | --- |
| 节奏 | BPM、Beat positions |
| 频谱 | STFT、Mel Spectrogram |
| 音色 | MFCC、Spectral centroid |
| 和声 | Chroma、Key（调性） |
| 能量 | RMS、Loudness（LUFS） |

##### 推荐工具链

*   **C++**
    *   `Essentia` ⭐⭐⭐⭐⭐（业界标准）
    *   `aubio`
*   **Python（用于原型）**
    *   `librosa`
    *   `madmom`（节拍非常强）

👉 建议：  
**Python 做算法验证 → C++ 重写核心路径**

* * *

#### 3️⃣ 特征存储

每首歌生成一个结构化索引：

```json
{
  "bpm": 128.3,
  "beats": [0.45, 0.92, 1.38, ...],
  "key": "8A",
  "mfcc_mean": [...],
  "chroma": [...],
  "energy_curve": [...]
}
```

存储方式：

*   SQLite（简单、可靠）
*   LMDB / RocksDB（大库）

* * *

### 阶段 2：相似度匹配 + 自动选歌

#### 4️⃣ 相似度模型

不是简单“BPM 差值最小”，而是**多维向量距离**：

```text
distance =
  w1 * |BPM1 - BPM2| +
  w2 * KeyDistance +
  w3 * MFCCDistance +
  w4 * EnergyTrendDistance
```

*   BPM：归一化差值
*   Key：**Circle of Fifths / Camelot Wheel**
*   MFCC：cosine / euclidean
*   Energy：DTW（动态时间规整）

👉 可以做成：

*   **规则系统（先）**
*   **ML 排序模型（后）**

* * *

#### 5️⃣ Transition Point 选择（进出点）

不是整首歌都适合混：

*   进点：低能量、节奏清晰
*   出点：break、drop 前

常见策略：

*   Energy valley
*   低频能量稳定区
*   Beat 强度下降区

* * *

### 阶段 3：实时 DJ Mixing Engine

#### 6️⃣ 节拍对齐 + Time Stretch

关键技术：

*   **Time-stretch without pitch change**
    *   WSOLA
    *   Phase Vocoder
    *   Rubber Band Library ⭐⭐⭐⭐

流程：

```
Song B BPM → 拉伸到 Song A BPM
Beat grid → 对齐最近的拍点
```

* * *

#### 7️⃣ 调性匹配（Key Match）

策略分级：

1.  同 key → 直接混
2.  相邻 key → OK
3.  差距大 → pitch shift（±1 semitone）

工具：

*   Rubber Band
*   SoundTouch

* * *

#### 8️⃣ 混音策略（不是简单 Crossfade）

专业 DJ 常用：

*   EQ swap（低频只保留一首）
*   高通 / 低通 filter sweep
*   渐进式混合（8 / 16 / 32 beats）

技术实现：

*   多轨 mixer
*   每轨独立 EQ / filter / gain automation

三、你一定会踩的坑（提前告诉你）
----------------

### 1️⃣ DSD 是个“反工程地雷”

*   不能直接分析
*   转 PCM 时：
    *   滤波质量影响音色
    *   CPU 开销大  
        👉 建议：

> 统一离线转 PCM 后再分析

* * *

### 2️⃣ 节拍检测不是万能的

*   古典 / 自由节奏 → BPM 不准
*   变速 / live → beat grid 会漂

解决：

*   置信度评分
*   fallback 到能量对齐

* * *

### 3️⃣ 时间拉伸 = 音质杀手

*   拉伸超过 ±6% 开始明显劣化
*   高频“水波纹”是常见 artefact

策略：

*   限制 BPM 差阈值
*   优先选 BPM 接近的歌

* * *

### 4️⃣ 实时系统比算法更难

*   音频 buffer underrun
*   lock-free queue
*   callback 内禁止分配内存

很多项目死在这里 😅
