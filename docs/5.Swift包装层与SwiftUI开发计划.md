# Swift 包装层与 SwiftUI 终端产品 — 开发计划

本文档对应主开发计划 [2.自动DJ混音引擎开发计划dev-plan.md](./2.自动DJ混音引擎开发计划dev-plan.md) 第六阶段第 2 项：「Swift 包装层与 SwiftUI 终端产品（当前优先）」。用于拆解任务、依赖与验收标准。

---

## 目标概述

- **Swift 包装层**：将 libautomix 的 C API 封装为 Swift 友好 API，便于 macOS/iOS App 集成。
- **SwiftUI 终端产品**：提供可运行的 macOS App，实现扫描曲库、生成播放列表、播放与基本控制，便于直观测试与迭代。

**依赖**：需先完成 libautomix 的构建并得到可链接的库（静态/动态）及头文件 `automix.h`、`types.h`。

---

## 阶段一：Swift Wrapper API 设计

**对应主计划**：设计 Swift wrapper API（AutoMixEngine class）

### 1.1 设计原则

- 单入口类封装引擎生命周期与操作，对应 C 的 `AutoMixEngine*`。
- 错误用 Swift `Error` 或 `Result` 表达，不再依赖 `automix_get_error()` 字符串。
- 资源（Engine、Playlist）用类/值类型 + 明确生命周期，避免 C 指针泄漏。

### 1.2 核心类型设计

| C API / 类型 | Swift 侧设计 |
|-------------|----------------|
| `AutoMixEngine*` | `AutoMixEngine` class，持有一个 `OpaquePointer` 或内部 C 句柄，`deinit` 中调用 `automix_destroy` |
| `PlaylistHandle` | `AutoMixPlaylist` 类或 struct，封装 `PlaylistHandle`，`deinit` 中 `automix_playlist_free` |
| `AutoMixError` | `enum AutoMixError: Error`，case 与 C 枚举一一对应 |
| `AutoMixPlaybackState` | `enum AutoMixPlaybackState` |
| `AutoMixTrackInfo` | `struct TrackInfo`（值类型），字段对应 C struct，注意 `path`/`key` 复制为 `String` |
| `AutoMixPlaylistRules` | `struct PlaylistRules`，可选字段用 Swift 可选类型 |
| `AutoMixTransitionConfig` | `struct TransitionConfig` |

### 1.3 方法映射（概要）

- **生命周期**：`init(dbPath: String)` → `automix_create`，`deinit` → `automix_destroy`。
- **扫描**：`scan(musicDir: String, recursive: Bool)`，`scan(musicDir:recursive:progress:)`（带进度回调）。
- **曲库**：`trackCount() -> Int`，`trackInfo(id:) -> TrackInfo?`，`searchTracks(pattern:) -> [Int64]`。
- **播放列表**：`generatePlaylist(seedTrackId:count:rules:) -> AutoMixPlaylist?`，`createPlaylist(trackIds:) -> AutoMixPlaylist?`。
- **播放控制**：`play(playlist:)`，`pause()`，`resume()`，`stop()`，`next()`（下一首，对应 C 的 `automix_skip`），`previous()`（上一首，对应 C 的 `automix_previous`；首曲时重头播放，否则跳回上一曲），`seek(seconds:)`（在当前曲目内定位到某一进度，对应进度条拖拽）。
- **状态**：`state`，`position`，`currentTrackId`；可选：`setStatusCallback` 的 Swift 封装。
- **音频**：`startAudio()`，`stopAudio()`，`render(buffer:frames:)` + `poll()`（用于自定义输出），`sampleRate`，`channels`。

### 1.4 交付物与验收

- [x] 设计文档或代码内注释：列出 `AutoMixEngine` 与相关类型的公开 API 列表。
- [x] 与现有 C API（`include/automix/automix.h`）的映射关系清晰，无遗漏核心能力。

## 阶段二：Swift Package（SPM）与 C 库集成

**对应主计划**：创建 Swift Package（SPM 支持）

### 2.1 包结构建议

```
Automix/
  Package.swift
  Sources/
    Automix/           # Swift 模块
      AutoMixEngine.swift
      Types.swift
      Errors.swift
      ...
  include/             # 或通过 systemLibrary 引用上层 include
    automix/
      automix.h
      types.h
```

- 若 libautomix 在仓库内由 CMake 构建：SPM 可通过 `systemLibrary` 引用预编译的 `.a`/`.dylib` 和 `include`，或在 Xcode 中通过「Link Binary With Libraries」引入。
- 若希望 SPM 独立构建 C 库：可评估 `target(type: .systemLibrary)` 或外部 CMake 产物拷贝到 SPM 可识别的路径。

### 2.2 头文件与模块映射

- C 头文件需可被 Swift 通过 `import Automix` 或 `@_implementationOnly import` 的 C 桥接头访问（或使用 `module.modulemap` 暴露 C 模块）。
- 确保 `automix.h` 中 `#include` 的路径在 SPM 构建时可用。

### 2.3 交付物与验收

- [x] `Package.swift` 可 `swift build`（或至少在 Xcode 中能编译并链接 libautomix）。
- [x] Swift 代码能调用 `automix_create`、`automix_destroy` 等至少 2～3 个 C 函数，无链接错误。

## 阶段三：类型桥接（C struct → Swift）

**对应主计划**：类型桥接（C struct → Swift struct）

### 3.1 必须桥接的类型（参考 `automix.h` / `types.h`）

- **只读/值类型**（C → Swift 拷贝）  
  - `AutoMixTrackInfo` → `TrackInfo`（`path`、`key` 转为 `String`，调用方不负责释放）。  
  - `AutoMixPlaylistRules` → `PlaylistRules`。  
  - `AutoMixTransitionConfig` → `TransitionConfig`。  
  - `AutoMixPlaybackState` → `PlaybackState`。  
  - `AutoMixError` → `AutoMixError`（Swift enum）。

- **回调相关**  
  - `AutoMixStatusCallback` → Swift 闭包 `(PlaybackState, Int64, Float, Int64) -> Void`。  
  - `AutoMixScanCallback` → Swift 闭包 `(String, Int, Int) -> Void`（当前文件、已处理数、总数）。

### 3.2 内存与所有权

- 从 C 取得的 `const char*`（如 `path`、`key`）在 Swift 侧复制为 `String` 后，不依赖 C 端生命周期。
- `automix_search_tracks` / `automix_playlist_get_tracks` 返回的 `int64_t*` 需在 Swift 封装内按 C API 约定释放（若 C API 要求调用方 free）。

### 3.3 交付物与验收

- [x] 所有上述类型在 Swift 中有对应实现，且可从 C 返回值/出参正确转换。
- [x] 至少一条端到端调用链使用这些类型（例如 `trackInfo(id:)` 返回 `TrackInfo` 并在 UI 中显示）。

## 阶段四：回调机制的 Swift 封装

**对应主计划**：回调机制的 Swift 封装（闭包/Combine/async-await）

### 4.1 状态回调（播放状态）

- C：`automix_set_status_callback(engine, callback, user_data)`。
- Swift：用 `Unmanaged` 或全局映射将 Swift 闭包作为 `user_data` 传入；在 C 回调中转调 Swift 闭包，注意线程（若 C 在音频线程调用，需派发到主线程再更新 UI）。
- 可选：暴露为 Combine `Publisher`（如 `playbackState: AnyPublisher<PlaybackState, Never>`）或 `@Published`，便于 SwiftUI 绑定。

### 4.2 扫描进度回调

- C：`automix_scan_with_callback(..., callback, user_data)`。
- Swift：封装为 `scan(musicDir:recursive:progress: (currentFile, processed, total) -> Void)`，同样处理 `user_data` 与线程（进度建议派发到主线程）。

### 4.3 可选：async-await

- 在 Swift 封装层提供 `func scan(...) async throws -> Int`，内部用 `withCheckedContinuation` 或 `Task` 包装带进度回调的 C 调用，在完成时 resume。
- 播放控制（play/pause/stop）若 C API 为同步，可直接在 async 方法中调用。

### 4.4 交付物与验收

- [x] 播放状态变化能在 Swift 闭包/Publisher 中收到，并能在 SwiftUI 中安全更新 UI。
- [x] 扫描进度能通过闭包或 async 接口获得，并在示例 UI 中显示进度。

## 阶段五：macOS 示例 App（SwiftUI + CoreAudio 验证）

**对应主计划**：macOS 示例 App（SwiftUI + CoreAudio 验证）

### 5.1 目标

- 验证「Swift 包装层 + 系统音频输出」全链路：创建引擎 → 扫描 → 生成播放列表 → `automix_start_audio` 播放。
- 不追求完整产品级 UI，以最小可验证为准。

### 5.2 功能清单（最小集）

- [x] 选择数据库路径（或使用默认路径）。
- [x] 选择音乐目录并触发扫描（带进度显示），扫描完成后显示曲库数量。
- [x] 从曲库选一首作为 seed，生成 N 首的播放列表并显示列表。
- [x] 播放/暂停/停止/上一首/下一首，以及通过进度条定位到当前曲目任意位置（seek）；并显示当前播放状态与进度。
- [x] 使用引擎内置 CoreAudio 输出（`automix_start_audio`），无需自定义 `render`/`poll`。

### 5.3 技术要点

- 主线程定期 `automix_poll()`（若使用内置音频则可能由引擎内部处理，需对照 C 实现确认）。
- 确保状态回调在主线程更新 SwiftUI，避免崩溃与警告。

### 5.4 交付物与验收

- [x] macOS App 可运行，能完成一次「扫描 → 生成列表 → 播放 → 暂停/停止/上一首/下一首/进度条定位」全流程。
- [x] 音频由 libautomix 输出，无二次解码或额外音频管线。

## 阶段六：面向终端用户的 SwiftUI 产品

**对应主计划**：面向终端用户的 SwiftUI 产品（便于直观测试与迭代）

### 6.1 定位

- 在示例 App 基础上增强为「可日常使用」的终端产品：更好浏览、筛选、播放体验，便于长期直观测试与迭代。
- 仍以 macOS 为先，iOS 可作为后续扩展。

### 6.2 功能增强（建议范围）

- [ ] **曲库浏览**：列表/网格展示曲库，支持按路径/ BPM/调性筛选与排序。
- [ ] **播放列表**：展示当前播放列表、顺序、下一首；支持从曲库拖拽或「从当前曲目生成后续列表」。
- [ ] **播放控制**：进度条（可拖拽定位到当前曲目任意位置并 seek）、上一首/下一首、音量（若引擎暴露）；当前曲目与下一首信息、过渡倒计时（若 C API 支持）。
- [ ] **设置**：数据库路径、默认扫描目录、过渡参数（crossfade beats、EQ swap 等）对应 `AutoMixTransitionConfig`。
- [ ] **稳定性与反馈**：错误提示（扫描失败、播放失败）、加载状态、空状态文案。

### 6.3 非功能

- [ ] 大曲库（百首量级）下列表不卡顿（懒加载/分页）。
- [ ] 扫描时 UI 可响应，不阻塞主线程（已由阶段四的异步/回调保证）。

### 6.4 交付物与验收

- [ ] 终端用户可完成：添加曲库 → 浏览 → 生成并播放列表 → 日常控制，无需改代码。
- [ ] 作为主开发计划中「直观测试与迭代」的载体，后续性能优化与功能扩展可基于此 App 验证。

     ↓

     ↓
